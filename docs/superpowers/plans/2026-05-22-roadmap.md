# leaStudios Email Templates — Improvements Roadmap

**Goal:** Land all ten improvement items from the 2026-05-22 code review.

**Approach:** One detailed plan document per phase, executed and committed independently. Each phase ends at a green-tests, clean-lint, shippable state. This roadmap defines the scope and order; each phase plan has the code-level steps.

**Order rationale:**

1. **Highest user value first** (preview, send-test) — visible win, no dependencies.
2. **Quick hygiene wins next** (PHPStan level, cache-busting) — low risk, prevent rot.
3. **Behind-the-scenes deliverability and ops** (plain-text alt body, send log).
4. **Cosmetic polish** (theme variants).
5. **Architectural refactors** (escape-contract, type registry) — these unblock #8.
6. **DX surface** (WP-CLI).
7. **Compliance** (unsubscribe) — biggest single piece; its own session.

---

## Phase 1 — Per-email-type preview + send-test-email

**Why:** Today the preview only renders the branding wrapper. Admins can't see how a `payment_receipt` actually looks before going live, and can't send themselves a test without faking a Stripe event.

**Scope:**
- New AJAX handler `wp_ajax_leastudios_email_templates_preview_type` rendering a chosen `Email_Type` with sample merge-tag context.
- New AJAX handler `wp_ajax_leastudios_email_templates_send_test` calling `Email_Sender::send` with sample context to a chosen address.
- UI buttons inside each accordion section in the Email Types tab.
- Sample-context helper so default tag values are realistic.

**Acceptance:** From `wp-admin → Email Templates → Email Types`, expand any type, click Preview → see the rendered HTML in an iframe; click Send Test → email lands at the chosen address.

**Detail plan:** `2026-05-22-phase-1-preview-and-send-test.md`

## Phase 2 — PHPStan level 7 + asset cache-busting

**Why:** Quick hygiene wins. Codebase is small enough to push past level 6, and a CSS-only fix shouldn't need a plugin version bump.

**Scope:**
- Bump `phpstan.neon` `level: 6` → `level: 7`, fix resulting errors (most likely around `array<string, mixed>` payloads in `Payment_Email_Listener` / `Payment_Data_Resolver`).
- Use `filemtime( LEASTUDIOS_EMAIL_TEMPLATES_DIR . '<asset>' )` for the `wp_enqueue_*` version arg.

**Acceptance:** `composer phpstan` clean at level 7; admin-page CSS hash visibly changes when only the CSS file is edited.

**Detail plan:** `2026-05-22-phase-2-hygiene.md`

## Phase 3 — Plain-text alternative body

**Why:** All transactional emails go out as HTML-only. Gmail's spam scoring and "show original" prefer `multipart/alternative` with a text version. Improves deliverability with no UX downside.

**Scope:**
- New `src/Email/Plain_Text_Generator.php` — convert HTML to readable plain text (drop scripts/styles, preserve links as `text (url)`, collapse whitespace).
- Hook `phpmailer_init` to set `$mail->AltBody` from the HTML body for transactional sends.
- Decide: only apply to our own transactional sends, or to every `wp_mail` HTML email? (Recommendation: every HTML email, since the branding wrapper already touches all of them.)

**Acceptance:** A multipart/alternative email arrives at Gmail/Outlook with both an HTML and a readable text part. Unit tests cover the converter (heading-to-line, link extraction, list bullets, table flattening).

**Detail plan:** `2026-05-22-phase-3-plain-text.md`

## Phase 4 — Send log / debug panel

**Why:** When a customer says "I didn't get my receipt", today there's no way to tell whether `wp_mail` returned true. The `leastudios_email_templates_email_sent` action already exists — just needs persistence and a viewer.

**Scope:**
- New table `wp_leastudios_email_templates_log`: id, type, recipient, subject, status (sent/failed), created_at, error (nullable).
- New `src/Database/Email_Log_Repository.php` and a migration registered on activation.
- Hook `leastudios_email_templates_email_sent` to record each row.
- New "Email Log" submenu page under "Email Templates" with paginated list + filters by type/status/date.
- Retention: prune rows older than N days (default 30) via a daily cron.

**Acceptance:** Trigger a transactional email; row appears in the log. Filter by `payment_failed`; only failures show.

**Detail plan:** `2026-05-22-phase-4-send-log.md`

## Phase 5 — Theme variants / dark-mode styles

**Why:** `templates/email/base.php` is solid for one look. Cheap to add variants.

**Scope:**
- Refactor `templates/email/base.php` to read theme tokens from a small array (background, body bg, text, muted, etc).
- Two new theme presets: `modern-light` (current), `modern-dark`.
- Add `@media (prefers-color-scheme: dark)` styles inline in `<head>` so dark-mode-aware clients adapt.
- Branding settings: dropdown for theme variant; persists to `leastudios_email_templates_branding[theme]`.
- Manual visual verification across Gmail web, Apple Mail, Outlook (no automated test for visual output).

**Acceptance:** Switching the theme dropdown changes the rendered preview without other code changes.

**Detail plan:** `2026-05-22-phase-5-theme-variants.md`

## Phase 6 — Per-tag raw/escape contract

**Why:** Today every merge tag value is HTML-escaped. Tags that should hold HTML (e.g. `{order_table}`, `{cta_button}`) can't render. Sets up Phase 7 (third-party tag registration).

**Scope:**
- Change `Email_Type::available_tags()` to return `[tag => ['description' => …, 'escape' => 'html'|'raw'|'url']]` (default html for back-compat).
- Update `Merge_Tag_Replacer::replace_html` to look up the escape mode per tag (with a default of `html`).
- New tests for each escape mode.
- Document the contract in the public-extension-points block.

**Acceptance:** A tag declared `'escape' => 'raw'` with HTML payload renders unescaped; one declared `'escape' => 'url'` survives `esc_url`; default tags still escape as before (no regression on existing tests).

**Detail plan:** `2026-05-22-phase-6-escape-contract.md`

## Phase 7 — Email-type registry (third-party extensibility)

**Why:** `Email_Type` is a closed enum. Plans like `leastudios-forms` sending its own confirmation emails through this plugin can't register a new type today.

**Scope:**
- New `src/Email/Email_Type_Registry.php` with an interface `Email_Type_Definition` (label, default_subject, default_body, available_tags).
- Built-in cases register themselves via `Plugin::init` on a new `leastudios_email_templates_register_types` action.
- `Settings_Page` iterates the registry instead of `Email_Type::cases()`.
- `Email_Sender::send` accepts `string $type_id` rather than `Email_Type` enum (or both — back-compat shim).
- All existing tests pass unchanged.

**Acceptance:** A test fixture plugin can register `my_plugin_welcome` via the action and have it appear in the Email Types tab with editable subject/body.

**Detail plan:** `2026-05-22-phase-7-type-registry.md`

## Phase 8 — WP-CLI subcommand

**Why:** Suite convention is to expose CLI for every plugin (see `wp-content/plugins/TESTING.md`). Useful for support/debugging.

**Scope:**
- New `src/CLI/Commands.php` registered only when `WP_CLI` is defined.
- Subcommands:
  - `wp leastudios-email-templates preview <type>` — print rendered HTML to stdout.
  - `wp leastudios-email-templates send-test <type> <email>` — send a real sample to the address.
  - `wp leastudios-email-templates list-types` — show all registered types.
- Requires Phase 7's registry to enumerate types.

**Acceptance:** All three commands work against the local Herd site; help text accurate.

**Detail plan:** `2026-05-22-phase-8-wp-cli.md`

## Phase 9 — Unsubscribe / suppression support

**Why:** Even for transactional mail, audits look for a working opt-out. Largest single piece — own session.

**Scope:**
- New table `wp_leastudios_email_templates_suppressions`: id, email, token, suppressed_at, source.
- New `src/Subscription/Unsubscribe_Manager.php`: token gen (signed, single-use-ish), verify, suppress, is_suppressed.
- New `src/REST/Unsubscribe_Controller.php`: `GET /wp-json/leastudios-email-templates/v1/unsubscribe?token=…` — landing page confirming opt-out.
- Global merge tag `{unsubscribe_url}` injected by `Merge_Tag_Replacer` when context contains a recipient.
- `Email_Sender::send` skips delivery (and logs the skip) when recipient is suppressed.
- Branding footer: option to auto-append "Unsubscribe" link.
- Per-type override: some emails (refunds, payment-failed) can be marked transactional-required and bypass suppression.

**Acceptance:** Visit unsubscribe URL → email recorded as suppressed → next non-required email to that address skipped → re-subscribe flow available via REST or admin.

**Detail plan:** `2026-05-22-phase-9-unsubscribe.md`

## Cross-cutting rules

- **TDD**: every phase. RED, watch fail, GREEN, refactor.
- **One commit per phase minimum**; multiple commits within a phase are fine and encouraged.
- **No backwards-compat shims** for code that hasn't shipped (per project CLAUDE.md). Phase 7's `Email_Sender::send` signature change is the only place this matters — current callers all live in this plugin.
- **No new fallbacks/workarounds without written approval** (per project CLAUDE.md).
- **Translator comments** on any new `__()` strings containing tokens.
- **Run `composer lint && composer test`** at the end of each phase. Don't move on until both are clean.
